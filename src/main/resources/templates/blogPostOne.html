Tomcat

  • The first question that we should answer is, what exactly is Tomcat? Before we can exactly what Tomcat is, we need to provide a little background.
  • When a web server receives a request, it first determines whether the resource being requested is static or dynamic in nature. A static resource is one that is identical no matter how many times, or in what order, that resource is requested. These static resources are the HTML and css pages
  • The web server handles the request for a static resource by simply fetching the identical resources from the appropriate path in which its static resides, and returning those bytes to the user.
  • A dynamic resource is a resource that uses its content based on various factors specific to the user making the request ( call to a database). Most web servers are tuned to deliver blazing fast request handling for static resources. However, they require additional help when dealing with dynamic resources. This help is typically provided by an additional software component that is registered with the web server, called a Servlet Container(more on servlet containers later).
  • When a request comes in that is directed at a dynamic resource(database), the web server simply hands off that request to its associate servlet container. The container then springs into action, invokes the appropriate servlet that delivers the requested dynamic resource. This servlet is responsible for generating the response. The generated response is returned by the servlet container to the web server which, in turn, returns it to the requesting user.
  • A servlet in its most basic form is a Java class that acts as a dynamic web resource. In other words, it can be the target of a client browser's request. Java has a whole API dedicated to creating servlets,which can be found HERE

Servlet Container

  • Ok so with all that background behind us, we can now ask what exactly is Tomcat? Well, Tomcat is a classified as a Servlet Container,that is, an environment which servlets can live and prosper. As a container it provides a lot of administrative support to servlets, allowing programmers to focus on the core application logic without having to bother with any low lever specifics, like class loading.
  • The primary responsibility of a container is to process an incoming request and to generate a corresponding response that is then returned to the client.

What is a Container?

  • Servlets don't have a main() method, they are under the control of another Java application called a container. Tomcat is an example of a container. When our web server application (like Apache) gets a request for a servlet, the server hands the request not to the servlet itself but to the container, in which the servlet is deployed. It's the container that gives the servlet the HTTP request and response, and it's the container that calls the servlet's methods.

Conclusion

  • Tomcat's main job is to create and handle the lifecycle and mapping of the Java Servlets. This might seem a little confusing right now, but it will make more sense once we start creating with Tomcat
